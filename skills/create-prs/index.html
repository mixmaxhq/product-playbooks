How to use

/create-prs

Or say "check before PR", "prepare PR", or "pre-flight checks" in any Claude Code conversation.

What it does

Runs pre-flight checks on your code changes, fixes common CI failures (lint, formatting, types), and creates clean PRs with proper descriptions. Encodes lessons learned from real Mixmax CI pipelines to catch issues before they reach GitHub Actions.

Workflow

1 Run pre-flight checks — lint, Prettier, TypeScript, and tests on all changed files
2 Fix issues — auto-fix lint errors, formatting, and type problems
3 Verify tests pass — run affected test files locally before pushing
4 Clean commits — squash if needed into conventional commit(s) with lowercase scope
5 Push and create PR — with description, links, deployment checklist, and test plan
6 Cross-link — link companion PRs if changes span multiple repos

Common CI failures this catches

Scope must be lowercase — feat(pg-963) not feat(PG-963). Commitlint enforces scope-case: lower-case.

Use React.Fragment — <React.Fragment> not <>. The react/jsx-fragments rule requires the long form.

No console statements — don't commit files with console.log, especially helper scripts. Keep dev-only tools untracked.

Run Prettier — CI will fail on formatting even if ESLint passes locally. Always run npx prettier --write on changed files.

Non-null assertions — array element access like arr[i] returns T | undefined. Use arr[i]! when you know the array is non-empty.

Multi-repo PRs

  • Create the simpler/dependency PR first
  • Note "companion PR to follow" in the first PR description
  • Create the second PR and link back using owner/repo#number
  • Update the first PR description to cross-link

Dev-only code

Remove before PR — local helper scripts, query param overrides, debug code, hardcoded test values.

Keep outside repo — if dev tooling is useful, add to .gitignore or keep untracked.
