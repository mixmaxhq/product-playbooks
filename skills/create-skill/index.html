/create-skill

Create a new Claude Code skill and publish it to the product-playbooks repo.

Install

Download the skill file and add it to your Claude Code skills directory:

mkdir -p ~/.claude/skills/create-skill && \
curl -Lo ~/.claude/skills/create-skill/SKILL.md \
  https://raw.githubusercontent.com/mixmaxhq/product-playbooks/main/skills/create-skill/SKILL.md

Restart Claude Code after installing. The skill will be available as /create-skill.

Or view SKILL.md on GitHub and copy it to ~/.claude/skills/create-skill/SKILL.md

How to use

/create-skill

Or say "create a skill for [topic]" in any Claude Code conversation.

What it does

Creates a new Claude Code skill end-to-end: writes the SKILL.md file, installs it locally, generates a documentation page, updates the product-playbooks index, and pushes to GitHub so it's live on the site.

Workflow

1 Understand the skill — what it does, when it triggers, what rules it follows
2 Write SKILL.md — create the skill file following format conventions
3 Install locally — save to ~/.claude/skills/<name>/SKILL.md
4 Create site page — add documentation page to product-playbooks repo
5 Update index — add card to the PM Skills section
6 Push to GitHub — commit and push so it's live on the site

SKILL.md format

---
name: skill-name-in-kebab-case
description: >
  Multi-line description of when to use the skill.
  Include trigger phrases the user might say.
---

# Skill Title — Skill

## What this does
One paragraph explaining what the skill produces.

## Workflow
Numbered steps — what happens when the skill runs.

## [Domain-specific sections]
Structure, rules, templates, or conventions.

Rules for writing good skills

Name — kebab-case, short, descriptive (e.g. gtm-brief, release-notes)

Description is the trigger — include exact phrases users might say. Be generous with triggers but specific about what the skill does.

Hardcode references — include specific Notion page IDs, repo URLs, parent pages. These make a skill autonomous vs. a generic prompt.

Encode lessons learned — if born from a manual workflow, capture the corrections that happened (e.g. "don't use internal jargon"). These are the most valuable parts.

One skill, one job — don't combine multiple workflows into one skill. Keep it focused.

No ephemeral content — don't include specific dates, ticket numbers, or temporary state. Skills should be durable.

Each skill is a directory — ~/.claude/skills/<name>/SKILL.md. The directory name is the identifier, the file is always SKILL.md.
